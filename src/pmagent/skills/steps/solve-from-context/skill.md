# Skill: 基于上下文定义问题

## Description

**先把问题空间定义清楚，再进入方案设计，避免过早被某个解法锁死。**

## 触发条件

- 用户希望 Agent 先充分理解问题，再进入方案设计
- 用户明确表示不想先讨论解决方案，而是先提供背景、事实、约束
- 用户需要 Agent 先定义清楚问题空间，避免过早收敛到某个解法

## 核心原则

- **只采集问题，不采集方案**：提问只能指向目标、现状、约束、证据、边界，禁止向用户索要解决方案。
- **先建模问题空间，不负责最终解法**：本 skill 只负责把问题定义清楚，不负责给出最优方案。
- **优先客观输入**：事实、数据、案例、现有流程、失败记录，优先级高于用户偏好。
- **AI 保持独立性**：即使用户带有明显倾向，Agent 也不能直接顺着写，必须基于上下文重新判断。
- **区分事实与推断**：输出中明确哪些来自用户输入，哪些是 Agent 的分析与假设。

## 适用场景

- 用户知道问题很复杂，但不想一开始就把 AI 锁死在某个方案里
- 做方案设计、产品方向判断、流程优化、需求拆解、策略选择前的问题澄清
- 作为后续方案生成类 skill 的前置步骤

## 不适用场景

- 用户已经明确要求按指定方案执行
- 任务目标是整理、润色或复述用户已有方案
- 高风险场景下必须由用户先给定不可更改的策略边界

## Agent 行为约束

- 禁止问：
  - "你觉得应该怎么做？"
  - "你倾向哪个方案？"
  - "你希望我给出哪些解法？"
  - 任何要求用户提前设计交互、流程、页面、技术方案的问题
- 可以问：
  - 目标是什么？
  - 当前发生了什么？
  - 谁受影响？
  - 现有做法是什么？
  - 有哪些硬约束？
  - 你已经观察到哪些证据？
  - 什么算成功，什么绝对不能发生？
- 如果用户主动给出方案偏好：
  - 记录为"用户倾向"，但不要直接采纳
  - 追问该倾向背后的事实依据，而不是沿着方案细化

## 执行步骤

### Step 1: 声明工作方式

开场先对齐方法：

- 说明本轮先收集客观上下文，不提前讨论方案
- 告知用户：Agent 会在信息足够后独立给出推荐，而不是复述用户设想

### Step 2: 建立问题空间

按以下顺序提问与整理，每次优先问最缺的一项，一次只追 1 个关键问题：

1. **目标**
   - 要解决的核心问题是什么？
   - 这件事为什么现在要解决？
2. **对象**
   - 谁是受影响最强的人？
   - 谁是决策者、使用者、付费者、执行者？
3. **现状**
   - 用户今天如何完成这件事？
   - 当前流程、路径、工具、角色分工是什么？
4. **痛点与证据**
   - 具体卡在哪里？
   - 有哪些数据、案例、投诉、失败记录、观察样本？
5. **约束**
   - 时间、资源、技术、组织、品牌、合规约束分别是什么？
6. **边界**
   - 什么属于本题，什么不属于？
   - 哪些前提不能改？
7. **成功标准**
   - 哪个结果能证明问题被解决？
   - 哪些负面结果必须避免？

### Step 3: 输出问题定义

在进入方案前，先产出一版结构化问题定义并回显给用户确认。至少包含：

```markdown
## 问题定义
- 目标：
- 对象：
- 当前现状：
- 核心痛点：
- 已有证据：
- 关键约束：
- 明确边界：
- 成功标准：

## 信息缺口
- 还缺什么：
- 缺口是否阻塞求解：
```

如果仍缺关键事实，继续追问事实问题；不要提前输出方案。

### Step 4: 标记用户倾向与假设

如果用户主动给出解法、偏好或方向，单独归档，不并入事实层：

- `用户倾向`：用户明确偏好的方向、解法、形式
- `工作假设`：当前尚未被证据证明但暂时成立的判断

要求：

- 不把用户倾向写成事实
- 不把工作假设写成既定结论
- 如果某个倾向缺少依据，明确标记为"待验证"

### Step 5: 以问题定义格式输出

默认按以下结构输出，作为后续方案生成类 skill 的标准输入：

```markdown
## 问题定义
- 核心目标：
- 目标对象：
- 当前现状：
- 核心痛点：
- 已有证据：
- 关键约束：
- 明确边界：
- 成功标准：

## 事实
- 已确认事实 1：
- 已确认事实 2：

## 用户倾向
- 倾向 1：
- 倾向背后的依据：
- 当前状态：已证实 / 待验证 / 无依据

## 假设
- 假设 1：
- 为什么暂时这样假设：

## 信息缺口
- 还缺什么：
- 缺口是否阻塞下一步：
```

## 质量检查

输出前自检：

- 我问的问题是否都在采集事实，而不是让用户替我设计方案？
- 我是否把用户偏好误当成事实？
- 我是否把事实、倾向、假设分开写清楚了？
- 我是否清楚标出了假设和不确定性？

## 常见失误

- 把"你想怎么做"伪装成澄清问题
- 用户一提某个方案，Agent 就顺着做细化
- 没有先定义成功标准，就直接开始出主意
- 输出看似客观，实则把用户偏好混进了事实层



## Step Contract (Protocol Boundary)

- **Reads**: Current workspace `Requirement.md`, `workspace-summary.md`, and the research/context/decisions/PRD files explicitly required by this step.
- **Writes**: This step's target artifact directory. If current state, conclusions, risks, or delivery status changes, update `workspace-summary.md`.
- **May mutate**: Files in the current workspace that are directly owned by this step.
- **Must not mutate**: Unrelated project/workspace files; PRD canonical content or project/global memory without user confirmation.
- **Required user confirmation**: Scope, risks, success metrics, PRD changes, project-level sync, and global memory deposition.
- **Handoff**: End by stating downstream input artifacts, confirmed conclusions, open questions, and conclusions downstream steps must not re-litigate.
