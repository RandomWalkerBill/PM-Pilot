# Skill: 第三方挑战 PRD

## Description

**用结构化挑战把当前 PRD 从“写出来”推进到“经得起质疑、适合继续交接”。**

## 触发条件

- PRD 写完后的质量审查阶段
- 0-to-1 模式 Step 7 / 迭代模式 Step 5
- write-prd 的 Step 3
- 用户主动要求"帮我挑战一下这个 PRD"

## 前置检查

1. 确认有可挑战的 PRD（已写完或基本成型）。
2. 确认上下文：是否有 Strategy Brief、历史 decisions 可供一致性校验。

## 核心原则

- **体验优先**：体验视角排第一，且权重最高。其他维度做得再好，用户用起来不对就是白搭。
- **每次只抛 1 个挑战**，等用户回应后再出下一个。
- **不放过弱回应**：用户回答不够有力时追问"这个论据不够，因为……你有更强的理由吗？"
- **挑战必须真实尖锐**，不降低强度来讨好用户。

## 挑战视角（按优先级排列）

### 视角 1：🟣 体验视角（Experience）— 最高优先级

> 这是最关键的视角。不是问"用户要不要"，而是深入到"用起来感觉怎么样"。

**挑战方向：**

- **首次体验**："用户第一次接触这个功能时，前 30 秒会发生什么？能不能不看说明就上手？"
- **核心路径感受**："用户完成主任务的操作路径有几步？每一步用户的心理状态是什么？有没有犹豫、困惑、等待的时刻？"
- **情绪曲线**："用户从打开到完成，情绪是上升的还是下降的？有没有'哇'的时刻？有没有'烦'的时刻？"
- **与现有方案对比**："用户现在怎么做这件事？你的方案在体验上具体好在哪里？好多少？值得用户切换吗？"
- **边界体验**："网络差的时候呢？数据很多的时候呢？用户犯错的时候呢？这些场景下体验会降级到什么程度？"
- **体验一致性**："这个功能的交互模式与产品其他部分一致吗？用户会不会觉得'这不像同一个产品'？"
- **可以更好吗**："如果不考虑工程成本，体验上你最想改什么？为什么现在没做？这个取舍合理吗？"

**输出要求：**
- 必须产出一段**用户体验走查**（Experience Walkthrough）：用第一人称模拟用户从触达到完成的完整感受
- 标注体验亮点（😊）和体验风险（😰）

### 视角 2：🔴 用户价值视角（User Value）

- "用户真的需要吗？痛点够痛吗？是'有点烦'还是'完全卡住'？"
- "目标用户定义清晰吗？是不是在试图讨好所有人？"
- "用户为什么要从现在的方案切换到你的？迁移成本多大？"
- "如果用户只用一次就走了，最可能的原因是什么？"

### 视角 3：🔵 工程可行性视角（Engineering）

- "技术可行吗？有没有看起来简单但实现很复杂的需求？"
- "依赖是否明确？有没有 blocking point？"
- "成本可控吗？MVP 范围合理吗？"
- "性能/稳定性有风险吗？"

### 视角 4：🟠 业务视角（Business）

- "ROI 合理吗？投入产出比算得过来吗？"
- "与现有产品/功能冲突吗？会蚕食已有业务吗？"
- "优先级合理吗？为什么不先做其他需求？"
- "上线 3 个月后如果失败了，最可能的原因是什么？"（Pre-Mortem）

### 视角 5：🟤 历史一致性视角（Consistency）— 迭代模式特有

> 仅在已有项目（迭代模式）中执行。0-to-1 无历史可查时跳过。

- "与已有 Strategy 的方向一致吗？如果偏离了，是 Strategy 该更新还是 PRD 该调整？"
- "与已有 decisions 矛盾吗？如果矛盾了，旧决策是否需要废弃？"
- "与已有功能的一致性如何？会不会造成产品内'两套逻辑'？"

## 执行步骤

### Step 1: 体验走查（必做）

Agent 以用户第一人称视角，走一遍 PRD 描述的核心流程，输出：

```markdown
### 体验走查

> 我是 [目标用户]，我要 [完成的任务]。

1. **触达**：我 [怎么发现/进入这个功能]……
   - 感受：[清晰/困惑/期待/...]
2. **理解**：我看到 [界面/信息]，我理解这是 [...]……
   - 感受：[一看就懂/需要想一下/完全不懂]
3. **操作**：我 [做了什么]……
   - 感受：[顺畅/犹豫/需要来回切换]
4. **反馈**：系统 [给了什么反馈]……
   - 感受：[有信心/不确定有没有成功]
5. **完成**：我完成了 [任务]，总体感觉 [...]

**😊 体验亮点**：
-

**😰 体验风险**：
-

**💡 体验可以更好**：
-
```

### Step 2: 逐视角挑战

按视角 1-5 的优先级顺序，每个视角抛出 1-2 个最尖锐的挑战。

- 每次只出 1 个挑战，等用户回应
- 用户回应后标记强度：✅ 坚实 / ⚠️ 需验证 / ❌ 薄弱
- 薄弱的追问，直到用户承认不确定或给出更强的理由

### Step 3: 输出挑战清单

```markdown
## 挑战清单

| # | 视角 | 挑战内容 | 用户回应 | 强度 |
|---|------|---------|---------|------|
| 1 | 🟣 体验 | ... | ... | ✅/⚠️/❌ |
| 2 | 🔴 用户价值 | ... | ... | ✅/⚠️/❌ |
| 3 | 🔵 工程 | ... | ... | ✅/⚠️/❌ |
| 4 | 🟠 业务 | ... | ... | ✅/⚠️/❌ |
| 5 | 🟤 一致性 | ... | ... | ✅/⚠️/❌ |

## 来源合规检查

> ⛔ 以下检查项必须在挑战清单输出时一并完成。

- [ ] Strategy 第 4 节（成功指标）的所有条目是否经用户确认？如有未确认的 AI 填入项，标记为 ❌ 并建议修正。
- [ ] Strategy 第 7 节（风险与对策）的所有条目是否经用户确认？如有未确认的 AI 填入项，标记为 ❌ 并建议修正。
- [ ] PRD 中引用的指标/风险是否与 Strategy 正文区一致？

```

### Step 4: 闭环

- 用户确认挑战清单
- 用户决定是否修改 PRD
- 如用户要求修改，修改完成后本 skill 结束
- 如有决策性结论，按 write-decision skill 落盘

> **⛔ GATE: 挑战闭环**
> 必须满足以下条件才能退出本 skill：
> - [ ] 体验走查已完成并展示给用户
> - [ ] 所有适用视角的挑战已展示（0-to-1 至少 4 视角，迭代至少 5 视角）
> - [ ] 用户已回复是否修改 PRD
> - [ ] 如用户要求修改，修改已完成
> 未通过此 Gate 不得进入下一步。


## Step Contract (Protocol Boundary)

- **Reads**: Current workspace `Requirement.md`, `workspace-summary.md`, and the research/context/decisions/PRD files explicitly required by this step.
- **Writes**: This step's target artifact directory. If current state, conclusions, risks, or delivery status changes, update `workspace-summary.md`.
- **May mutate**: Files in the current workspace that are directly owned by this step.
- **Must not mutate**: Unrelated project/workspace files; PRD canonical content or project/global memory without user confirmation.
- **Required user confirmation**: Scope, risks, success metrics, PRD changes, project-level sync, and global memory deposition.
- **Handoff**: End by stating downstream input artifacts, confirmed conclusions, open questions, and conclusions downstream steps must not re-litigate.
